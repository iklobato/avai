avai documentation

Everything you need to run avai with Docker — every command, every parameter, every environment variable. For the plain‑English overview, see the home page.

Overview

avai has two parts that share a single SQLite file:

  • • monitor — runs the collectors, enriches findings against threat‑intel sources, and asks an LLM for a verdict. Writes the database.
  • • dashboard — a read‑only Flask web UI on port 8765. Reads the database.

Both ship in one Docker image, iklob1/avai. The default command is the dashboard; override it to run the monitor. The monitor needs a Linux host (it reads /proc, sysfs, journald, systemd); the dashboard runs anywhere Docker does.

Quick start (60 seconds, no key)

Populate a database with one scan, then serve it. Works on any OS that runs Docker.

mkdir -p ~/.avai && cd ~/.avai

# 1. one scan (no LLM, no external calls — just inventory)
docker run --rm -v "$PWD":/data iklob1/avai \
  avai monitor --once --no-streaming --no-judge --no-enrich --db /data/avai.db

# 2. view it
docker run -d --name avai -p 8765:8765 -v "$PWD":/data iklob1/avai
open http://localhost:8765/

Run the dashboard

The dashboard needs no privileges — just a directory holding avai.db mounted at /data and a published port.

docker run -d \
  --name avai-dashboard \
  -p 8765:8765 \                # publish :8765
  -v "$PWD":/data \             # dir containing avai.db
  iklob1/avai                   # default CMD = avai dashboard
docker run -d \
  --name avai-dashboard \
  -p 8765:8765 \                # publish :8765
  -v "$PWD":/data \             # dir containing avai.db
  iklob1/avai                   # default CMD = avai dashboard
# PowerShell — single line
docker run -d --name avai-dashboard -p 8765:8765 -v ${PWD}:/data iklob1/avai

Custom port / DB path — append your own avai dashboard command:

docker run --rm -p 9000:9000 -v /var/lib/avai:/data iklob1/avai \
  avai dashboard --host 0.0.0.0 --port 9000 --db /data/custom.db

On a fresh empty dir the dashboard creates an empty schema and every panel renders empty until the monitor writes rows.

Run the monitor (Linux host)

The monitor reads host state, so it needs the host PID + network namespaces, a few capabilities, and the host filesystem bind‑mounted under /host (via HOST_PREFIX). Continuous, with the LLM judge + enrichment enabled. Add your keys with -e ANTHROPIC_API_KEY -e ABUSE_CH_AUTH_KEY. 

docker run -d --name avai-monitor --restart unless-stopped \
  --pid=host --network=host --user 0:0 \
  --cap-add SYS_PTRACE --cap-add NET_ADMIN --cap-add NET_RAW --cap-add DAC_READ_SEARCH \
  -e HOST_PREFIX=/host \
  -e DBUS_SYSTEM_BUS_ADDRESS=unix:path=/run/dbus/system_bus_socket \
  -e ANTHROPIC_API_KEY -e ABUSE_CH_AUTH_KEY \
  -v /proc:/host/proc:ro -v /sys:/host/sys:ro -v /etc:/host/etc:ro \
  -v /var/lib/dpkg:/host/var/lib/dpkg:ro \
  -v /var/lib/bluetooth:/host/var/lib/bluetooth:ro \
  -v /usr/share/applications:/host/usr/share/applications:ro \
  -v /lib/systemd:/host/lib/systemd:ro -v /usr/lib/systemd:/host/usr/lib/systemd:ro \
  -v /var/log/journal:/host/var/log/journal:ro -v /var/spool/cron:/host/var/spool/cron:ro \
  -v /run/systemd:/run/systemd:ro -v /run/dbus:/run/dbus:ro \
  -v /etc/machine-id:/etc/machine-id:ro -v /dev/mapper:/dev/mapper:ro \
  -v /home:/host/home:ro -v /root:/host/root:ro \
  -v "$PWD":/data \
  iklob1/avai \
  avai monitor --db /data/avai.db --interval 300
# Docker Desktop on macOS can't share /root, /home, etc. and only
# exposes its Linux VM — so this monitors the VM, not your Mac.
# For real Mac coverage use the native install (see note below).
docker run -d --name avai-monitor --restart unless-stopped \
  --pid=host --network=host --user 0:0 \
  --cap-add SYS_PTRACE --cap-add NET_ADMIN --cap-add NET_RAW --cap-add DAC_READ_SEARCH \
  -e HOST_PREFIX=/host \
  -v /proc:/host/proc:ro -v /sys:/host/sys:ro -v /etc:/host/etc:ro \
  -v "$PWD":/data \
  iklob1/avai \
  avai monitor --db /data/avai.db --interval 300 --no-streaming
# PowerShell. Docker Desktop exposes only its Linux VM (WSL2), not
# Windows — so this monitors the VM. Single line:
docker run -d --name avai-monitor --restart unless-stopped --pid=host --network=host --user 0:0 --cap-add SYS_PTRACE --cap-add NET_ADMIN --cap-add NET_RAW --cap-add DAC_READ_SEARCH -e HOST_PREFIX=/host -v /proc:/host/proc:ro -v /sys:/host/sys:ro -v /etc:/host/etc:ro -v ${PWD}:/data iklob1/avai avai monitor --db /data/avai.db --interval 300 --no-streaming
⚠️ macOS / Windows: the monitor only works fully on a Linux host. On Docker Desktop it sees the bundled Linux VM, not your real machine. To watch your actual Mac/PC, install natively: pip install 'avai-monitor[judge]' then sudo avai monitor (Linux/macOS).

Docker run flags (monitor)

Why each privilege flag is needed:

FlagWhy
--pid=hostso psutil sees the host's processes, not just the container's
--network=hostso network connections / listening ports are the host's
--user 0:0root inside the container — needed to read protected host files
--cap-add SYS_PTRACEread /proc/<pid> across processes (psutil)
--cap-add NET_ADMIN NET_RAWWi‑Fi (NL80211 netlink) + some sysfs network reads
--cap-add DAC_READ_SEARCHread root‑owned files (/etc/shadow, other users' crontabs) without UID juggling
--restart unless-stoppedkeep the monitor running across reboots

Bind mounts

With HOST_PREFIX=/host, collectors read /host/<path> instead of the container's own. Mount each host path you want covered read‑only under /host:

MountFeeds collector
/proc:/host/proc:roprocesses, network
/sys:/host/sys:rousb_devices, wifi_state
/etc:/host/etc:rofile_integrity, system_integrity
/var/lib/dpkg:/host/var/lib/dpkg:roinstalled_apps (Debian/Ubuntu)
/var/lib/bluetooth:/host/var/lib/bluetooth:robluetooth_devices
/{lib,usr/lib}/systemd:/host/…:rolaunch_items (unit files)
/var/spool/cron:/host/var/spool/cron:rolaunch_items (cron)
/var/log/journal:/host/var/log/journal:roauth_events, process_exec_events
/home:/host/home:ro, /root:/host/root:roper‑user dotfiles, user systemd units, browser extensions
/run/systemd, /run/dbus (native paths)systemctl / dmsetup talk to the host's systemd + D‑Bus
"$PWD":/datawhere avai.db is written

Mount only what you need — a missing mount just means that collector finds nothing; it never crashes.

Easiest is an .env file with --env-file:

cp .env.example .env   # fill in the keys you have
docker run -d --env-file .env … iklob1/avai avai monitor …

docker compose

One image, two services. Monitor writes ./data/avai.db; dashboard reads it.

x-avai-image: &avai-image
  image: iklob1/avai:latest

services:
  monitor:
    <<: *avai-image
    command: ["avai","monitor","--db","/data/avai.db","--interval","300"]
    user: "0:0"
    pid: host
    network_mode: host
    cap_add: [SYS_PTRACE, NET_ADMIN, NET_RAW, DAC_READ_SEARCH]
    env_file: [.env]
    environment:
      - HOST_PREFIX=/host
      - DBUS_SYSTEM_BUS_ADDRESS=unix:path=/run/dbus/system_bus_socket
    volumes:
      - ./data:/data
      - /proc:/host/proc:ro
      - /sys:/host/sys:ro
      - /etc:/host/etc:ro
      - /run/systemd:/run/systemd:ro
      - /run/dbus:/run/dbus:ro
      # … add the rest of the /host mounts as needed
    restart: unless-stopped

  dashboard:
    <<: *avai-image
    ports: ["8765:8765"]
    volumes: ["./data:/data"]
    restart: unless-stopped
mkdir -p data && cp .env.example .env
docker compose up -d
docker compose logs -f monitor

Recipes

One‑shot scan (no daemon)

docker run --rm --pid=host --network=host --user 0:0 \
  --cap-add SYS_PTRACE --cap-add DAC_READ_SEARCH -e HOST_PREFIX=/host \
  -v /proc:/host/proc:ro -v /sys:/host/sys:ro -v /etc:/host/etc:ro \
  -v "$PWD":/data iklob1/avai \
  avai monitor --once --no-streaming --db /data/avai.db

Keep LLM cost low

avai monitor --judge-model claude-haiku-4-5-20251001 \
             --judge-max-per-collector 20

Only new artifacts are judged and verdicts are cached, so steady‑state cost is near zero.

Debug / restrict enrichment

avai monitor --no-enrich                          # off entirely
avai monitor --enrich-only cisa_kev               # one source
avai monitor --enrich-only virustotal --enrich-only abuseipdb

sources: malware_bazaar urlhaus threatfox circl_hashlookup shodan_internetdb feodo_tracker osv cisa_kev nvd endoflife crtsh virustotal abuseipdb greynoise safe_browsing phishtank github_advisory

Bring your own LLM provider

avai monitor --judge-model gpt-4o-mini            # OpenAI (OPENAI_API_KEY)
avai monitor --judge-model ollama/llama3.1        # local, offline, free
avai monitor --judge-model gemini/gemini-1.5-pro  # Google

Reading findings without the dashboard

It's one SQLite file — query it directly for scripting or a headless server:

# dangerous + suspicious findings, newest first
sqlite3 -box data/avai.db "SELECT verdict, collector, substr(reasoning,1,60)
  FROM judgements WHERE verdict IN ('malicious','suspicious')
  ORDER BY created_at DESC LIMIT 20;"

# count by verdict
sqlite3 data/avai.db "SELECT verdict, count(*) FROM judgements GROUP BY verdict;"

# what the threat-intel sources said
sqlite3 -box data/avai.db "SELECT source, verdict_hint, substr(summary,1,70)
  FROM enrichment_evidence WHERE verdict_hint IN ('malicious','suspicious');"

Troubleshooting

Dashboard shows nothing / all panels empty

No data yet — run the monitor at least once against the same /data dir. On macOS the monitor in Docker only sees the Linux VM, so install natively for real Mac data.

No malicious/suspicious verdicts, only raw rows

The judge is off. Pass an LLM credential (-e ANTHROPIC_API_KEY=…) and don't use --no-judge.

An enricher isn't running

Keyed sources only start if their env var is set and non‑empty. Check the monitor log at startup — it prints enricher enabled: … / enricher disabled (token missing): … for each.

A collector reports rows=0 on Linux

Usually a missing bind mount — e.g. no launch_items without /lib/systemd mounted under /host. Mount the relevant path (see Bind mounts).

Check container health & logs 
docker logs -f avai-monitor
docker inspect avai-dashboard --format '{{.State.Health.Status}}'
docker exec avai-monitor du -h /data/avai.db   # DB growth / rotation
